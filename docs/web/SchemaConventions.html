Schema Conventions

The xml schema of xidgets and xidget script is designed to be flexible so that new users who are unfamiliar with the schema can write more verbose, readable xml, while veterans can code succinctly. Xidget script, in particular, is designed so that parameters can be specified in either an element or an attribute. Xidget configuration, on the other hand, tends to favor child elements because xidgets have so many parameters. I really dislike xml schemas where the majority of the schema is defined in attributes.

Although a script always has a context, in general scripting elements require that all required parameters be explicitly stated. This makes the script easier to understand. In other words, there are no scripting element parameters that assume the script context when omitted.

As an example, consider the add scripting element, which adds a child element to a parent element. The add scripting element requires a source and a target xpath. The source xpath identifies the elements which will be parented. The target xpath identifies the elements that will become the parent of the source elements. Both the source and the target parameters may be specified as either attributes or child elements. In addition, to support the most compact representation, one of the parameters may be specified in the text of the scripting element. So, all of the following examples have identical behavior:

<add> <source>$newBook"</source> <target>collection('library')/books/book[ title='The Lightness of Being']</target> </add>
<add source="$newBook" target="collection('library')/books/book[ title='The Lightness of Being']"/>
<add source="$newBook">collection('library')/books/book[ title='The Lightness of Being']</add>
<add target="collection('library')/books/book[ title='The Lightness of Being']">$newbook</add>

For more information on the add script, see Scripting Reference.
